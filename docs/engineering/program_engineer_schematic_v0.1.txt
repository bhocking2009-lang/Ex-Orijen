Ex Origine — Program Engineer Schematic (v0.1)
Type: Engineering Implementation Map
OMA Alignment: A1 → A5


1. Purpose

This document translates the Ex Origine vision and project wireframe into an engineering-facing schematic.

It defines the major program components, execution order, module responsibilities, data boundaries, and development priorities required to begin implementation.


2. Engineering Objective

Build a modular real-time simulation framework where:

- Systems are independently testable
- Runtime behavior is deterministic when required
- Data drives rules and configuration
- State changes are logged and traceable
- Emergent complexity arises from interacting subsystems

7. Simulation Update Order

Each simulation tick should follow this order:

1. Begin Tick
2. Read queued player/system inputs
3. Update Energy System
4. Update Information System
5. Update Entity System
6. Resolve Interactions
7. Update Environment System
8. Commit state changes
9. Write logs and diffs
10. End Tick


8. Data Ownership Rules

Energy System owns:
- Energy pools
- Transfer rules
- Energy availability

Information System owns:
- Signals
- Memory
- Pattern history

Entity System owns:
- Entity definitions
- Behavior state
- Action intent

Environment System owns:
- Spatial data
- Terrain/resource layout
- Environmental constraints

Interaction System owns:
- Cross-system resolution
- Final interaction outcomes

State Manager owns:
- Authoritative committed state


9. Engineering Constraints

- No system directly mutates another system's private state
- All cross-system changes pass through events or state commits
- No entity action executes without energy validation
- Every tick must be reproducible from seed plus input log
- All major rules must be configurable through data files
- Every system must support isolated unit tests


10. Minimum Viable Engineering Build

Version: v0.0.1

Goal:
Create a working simulation loop with one simple entity, one energy pool, and logged tick updates.

Required Features:
- Application starts
- Configuration loads
- Fixed tick loop runs
- Energy system updates
- Entity consumes energy
- State manager records changes
- Logger writes tick output
- Program exits cleanly

Not Required Yet:
- Graphics
- AI
- Complex environment
- Multiplayer
- UI
- Advanced physics
