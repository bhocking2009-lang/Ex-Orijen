# Ex Origine v0.1 Closeout Checklist

This checklist defines what it means to close the v0.1 loop. Copilot's draft PR establishes the architecture and MVB skeleton; this document is the independent verification layer before merging.

## Closeout Goal

v0.1 is complete when a clean checkout can build, run, and explain a small simulation loop from repository data files alone.

The acceptance question is:

> Can the world run for 100 ticks and produce explainable state changes?

## Required Checks

- Build the MVB executable from source with a C++17-compatible compiler.
- Run the default config for 10 ticks and confirm clean startup and shutdown.
- Run the default config for 100 ticks and confirm tick begin/end logging stays ordered.
- Build and run the unit test suite.
- Build and run the MVB integration test.
- Build and run the legacy smoke tests.
- Run the executable with a missing config path and confirm it exits with failure.
- Change `data/configs/simulation_config.json` values locally and confirm they affect runtime behavior.

## Expected MVB Behavior

- `data/configs/simulation_config.json` controls tick rate, max ticks, grid size, initial energy, and decay rate.
- `Application::init` fails fast when the config cannot be loaded.
- A non-positive `tick_rate_hz` falls back to 20 Hz instead of entering invalid scheduler math.
- A non-positive `max_ticks` config value is ignored in favor of the default unless the CLI supplies a valid value.
- The simulation publishes paired `TickBegin` and `TickEnd` events.
- Energy decays and entity actions consume energy through `EnergySystem`.
- Entity actions are gated by energy withdrawal.
- State history records one committed snapshot per tick.

## Verification Commands

Linux/macOS/Git Bash:

```sh
./scripts/verify_v0_1.sh
```

Windows PowerShell:

```powershell
.\scripts\verify_v0_1.ps1
```

The POSIX script uses `g++`. The PowerShell script uses `g++` when available and otherwise falls back to Visual Studio `cl.exe` through `VsDevCmd.bat`.

## Human Review Notes

Because v0.1 is an architectural skeleton, the important review is not visual polish. Reviewers should focus on:

- Does each system remain modular and replaceable?
- Are cross-system effects carried through events instead of private-state mutation?
- Does the runtime make configuration failures visible?
- Are replay/log/state hooks present for later causality inspection?
- Is the next v0.2 observer loop obvious from the current shape?

## v0.2 Handoff

Once v0.1 is merged, v0.2 should add the first visible observer loop:

- 2D grid state export.
- Energy heatmap data.
- Entity position/state projection.
- Pause, step, and run controls.
- Event log inspection.
